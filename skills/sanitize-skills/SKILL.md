## 背景

`skills/` は `gh skill install` を通じて他プロジェクトへ配布される共有ライブラリである。`SKILL.md` の例示や `examples/` のサンプルは、作成時に手元の実プロジェクトの内容をベースに書かれることが多く、特定の社内ドライブ名・サービスのアカウントID・絶対パス・他リポジトリ名がそのまま残ってしまうことがある。これらは配布先プロジェクトでは意味を持たないノイズになるだけでなく、意図しない情報開示にもなるため、共有前に検査して汎用化する。

## 対象範囲

- 引数でスキル名が指定された場合: `skills/<スキル名>/` 以下の全ファイル
- 指定がない場合: `skills/` 配下の全スキルディレクトリの `SKILL.md`・`examples/**/*.md`・`resources/**/*.md`

対象ファイルは Glob で列挙してから Read で読み込む。

## 検出パターン

以下は機械的パターンの手がかりに過ぎない。パターンに一致しても文脈上問題ない汎用的な技術例（HTTPステータスコード、公開APIの仕様、一般的なライブラリ名等）は誤って検出しないよう、都度「これは特定プロジェクトの実データか、汎用的な技術知識か」を文脈で判断すること。判断に迷うものは「要確認」として残し、機械的に削除・置換しない。

| パターン | 手がかり | 判断の視点 |
|---|---|---|
| 絶対パス | `/Users/...`、`/home/...`、`C:\...` | 特定端末・特定リポジトリへのパスか |
| 他リポジトリへの参照 | `github/<org>/<repo>` 形式の文字列やリポジトリ名の直書き | 現在のリポジトリと異なる実在プロジェクト名か |
| ラベル付きの長い数値ID | `Hub ID`、`Account ID`、`Workspace ID`、`Tenant ID`、`Sheet ID` 等の語 + 4桁以上の数字 | 実在のサービスアカウント・テナントを特定できる値か |
| 固有名詞 + 資産を表す語 | `「」`や`""`で囲まれた語 + ドライブ／シート／フォルダ／管理表／システム等 | 特定の社内資産名を指しているか |
| 命名規則的なファイル名 | 年月＋業務略称のパターン（例: `2026_06_xx`） | 実際の業務ファイル名か |
| 連絡先・内部URL | メールアドレス・電話番号・社内ダッシュボードのURL | 実在の連絡先・内部システムか |

## 手順

1. **対象ファイルの収集**: Glob で対象範囲のファイルを列挙する
2. **走査**: 各ファイルを Read し、上記パターンの手がかりを Grep で当たりをつけつつ、該当箇所の前後の文脈を読んで実データかどうかを判定する
3. **findings のまとめ**: 実データと判定した箇所ごとに「ファイル:行」「該当箇所の引用」「判定理由」「汎用化の提案」を記録する。要確認のものは「要確認」として理由とともに記録する
4. **レポートの書き出し**: 以下のテンプレートに従い、CLAUDE.md の規約に従って `.vscode/docs/plans/sanitize_skills_<対象名または all>_<YYYY-MM-DD>.md` に書き出す（ディレクトリが無ければ作成する）
5. **承認待ち**: レポートの概要をユーザーに伝え、置換内容に承認を得る。**承認前に SKILL.md や examples を書き換えてはならない**
6. **適用**: 承認された項目のみ Edit で置換する。プレースホルダーは元の文の文法・体言止め・記法（`語: 語` か文か）を保ったまま、内容だけを抽象化する
7. **再確認**: 置換後、対象ファイルに同じパターンが残っていないか Grep で再確認する
8. **完了報告**: 置換したファイルと件数、対応不要と判断した件数（理由付き）、要確認として残した件数を報告する

## レポートフォーマット

```markdown
# skills サニタイズ検査結果

| 項目 | 内容 |
|---|---|
| 対象 | <スキル名 or "skills/ 配下全体"> |
| 実施日 | <YYYY-MM-DD> |
| 検出件数 | 実データ: N件 / 要確認: N件 |

## 実データと判定した項目

### <ファイルパス>:<行番号>

- 該当箇所: `<引用>`
- 判定理由: <なぜ特定プロジェクトの実データと判断したか>
- 汎用化案: `<置換後の文言>`

## 要確認

### <ファイルパス>:<行番号>

- 該当箇所: `<引用>`
- 迷った理由: <汎用的な技術例の可能性がある等>

## 対応不要

- <スコープに含めたが実データと判定しなかった箇所とその理由>
```

## 汎用化の指針

- 具体的な社名・サービス名・識別子は、同じ品詞・同じ役割を持つ抽象的な語に置き換える（例: 実在のドライブ名 → `ドライブA`、実在のID → `xxxxxxx`）
- 元の例が伝えたい技術的・構造的なポイント（原則・手順の意図）は変えない。実データを抜いても説明が成立するように書き換える
- どうしても具体性がないと説明が成立しない場合は、完全に架空の一般的な名称（例: `sample-project`、`report.csv`）を使う

## 注意事項

- 誤検知を避けるため、機械的パターンへの一致だけで自動的に書き換えない。文脈判断と要確認への振り分けを必ず行う
- ユーザーの承認前に既存ファイルを変更しない
- frontmatter の `description` 等も検査対象に含める（実プロジェクト名がトリガー文に紛れ込んでいることがある）
