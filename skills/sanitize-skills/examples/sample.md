# sanitize-skills 適用例

架空のスキル `invoice-export` の `SKILL.md` に、別プロジェクトの実データが紛れ込んでいたケースを想定する。

## 検査対象（サニタイズ前）

```markdown
## 出力例

- 対象ファイルは `2026_05_invoice_batch`（格納先「経理精算システムSaaS」、テナントID `9081234`）
- 送信先は Slack Webhook ではなく社内通知システムのエンドポイント `notify-api.internal.acme-corp.jp` を使う
- レスポンスコードが `200` 以外の場合はリトライする
```

## 検出結果レポート（要約）

```markdown
# skills サニタイズ検査結果

| 項目 | 内容 |
|---|---|
| 対象 | invoice-export |
| 実施日 | 2026-07-06 |
| 検出件数 | 実データ: 3件 / 要確認: 0件 |

## 実データと判定した項目

### skills/invoice-export/SKILL.md:12

- 該当箇所: `2026_05_invoice_batch`（格納先「経理精算システムSaaS」、テナントID `9081234`）
- 判定理由: 年月＋業務略称の命名規則と、ラベル付きの長い数値ID（テナントID）が実在のサービスアカウントを特定できる
- 汎用化案: `file_x`（格納先「ドライブA」、テナントID `xxxxxxx`）

### skills/invoice-export/SKILL.md:13

- 該当箇所: `notify-api.internal.acme-corp.jp`
- 判定理由: 特定企業の内部ドメインを直接指しており、他プロジェクトでは意味を持たない
- 汎用化案: 具体的なホスト名を削除し、役割の説明に置き換える

## 対応不要

- skills/invoice-export/SKILL.md:14「レスポンスコードが `200` 以外の場合はリトライする」: HTTPステータスコードは一般的な技術知識であり、特定プロジェクトの実データではない
```

## 出力（承認後にサニタイズ）

```markdown
## 出力例

- 対象ファイルは `file_x`（格納先「ドライブA」、テナントID `xxxxxxx`）
- 送信先は Slack Webhook ではなく社内通知システムのエンドポイントを使う
- レスポンスコードが `200` 以外の場合はリトライする
```

## ポイント

- `200` のような一般的な技術知識はパターンに引っかからないよう「対応不要」に振り分け、誤って書き換えない
- ホスト名のように汎用的な言い換えでは元の意図が薄れる場合、具体的な値を消して役割の説明に言い換える（架空の値を無理に補わない）
- レポートを先に提示し、承認を得てから `SKILL.md` を書き換える
